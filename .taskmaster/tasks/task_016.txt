# Task ID: 16
# Title: Restructure Employee Data Model: Separate Master and Vacation Data
# Status: done
# Dependencies: 5, 14
# Priority: high
# Description: Refactor the current Employee model to split immutable master data and annual vacation data into separate tables, extend employee fields, and update the UI for improved clarity and management.
# Details:
1. **Backend Data Model:**
   - Create a new `Employee` table for master data: `id`, `title`, `first_name`, `last_name`, `position`, `email`, `birth_date`, `date_hired`, `federal_state`, `active`.
   - Create a `VacationAllowance` table: `id`, `employee_id` (FK), `year`, `annual_allowance`, `carryover_days`, `total_allowance`, `created_at`, `updated_at`.
   - Define a `FederalState` enum covering all 16 German states.
   - Write migrations to split existing employee data and migrate vacation-related fields to the new structure.
2. **API Layer:**
   - Update schemas, endpoints, and validation to support the new structure.
   - Ensure endpoints allow CRUD for both master and vacation data, and support filtering by year and employee.
3. **Frontend:**
   - Update TypeScript interfaces to match the new backend models.
   - Refactor UI components:
     - Employee list: compact view with name, position, date hired, status.
     - Employee detail: full master data and vacation data management, with clear separation.
     - Integrate federal state selection and display.
   - Ensure responsive design for desktop, tablet, and mobile.
4. **Data Migration:**
   - Implement scripts or migration logic to move existing data to the new structure without loss.
5. **Documentation:**
   - Document the new data model, API changes, and UI updates for future maintainability.

# Test Strategy:
- Run database migrations and verify data integrity: all employees have correct master and vacation data split.
- Test all CRUD operations for both Employee and VacationAllowance via API and UI.
- Confirm that the employee list displays only the required fields and is responsive.
- Verify that the detail view allows editing all master and vacation data, and that changes persist.
- Check that federal state selection works and is stored correctly.
- Perform regression tests to ensure no loss of functionality in employee management.
- Validate data migration by comparing pre- and post-migration records for accuracy.

## 3. VacationAllowance Model erstellen [done]
### Dependencies: 16.2
### Description: Erstelle ein neues VacationAllowance Model für die jährlichen Urlaubsdaten der Mitarbeiter.
### Details:
Erstelle backend/app/models/vacation_allowance.py mit folgenden Feldern:

- id: int (Primary Key)
- employee_id: int (Foreign Key zu Employee)
- year: int (Kalenderjahr, z.B. 2024)
- annual_allowance: int (Jahresurlaubsanspruch in Tagen)
- carryover_days: int (Übertragene Tage vom Vorjahr, default=0)
- total_allowance: int (Computed: annual_allowance + carryover_days)
- created_at: datetime (Erstellungszeitpunkt)
- updated_at: datetime (Letzte Änderung)

Constraints:
- Unique constraint auf (employee_id, year)
- Foreign Key Beziehung zu Employee
- Validierung: year >= 2020, annual_allowance >= 0, carryover_days >= 0

Relationships:
- Beziehung zu Employee Model (many-to-one)
- Employee sollte back_populates zu vacation_allowances haben
